## 設計ドキュメント作成（← ここからスタート）

> なぜ設計から始めるの？
> コードを書く前に「何を作るか」を言語化しておくと、途中で迷子になりません。
> 建築で言うと、工事を始める前に「設計図」を描くのと同じです。

**目標**：自分だけのタスク管理アプリのコンセプトを決め、設計図をドキュメントにまとめる

**進め方**：

1. Claude Code に以下を伝えてアイデアを壁打ちする（10〜15分）
   ```
   「私はタスク管理アプリを作ります。
   どんな人向けにすると面白いか、差別化できる機能のアイデアを5つ提案してください。」
   ```
2. 気に入ったアイデアを1〜2個選ぶ
3. Claude Code に「設計ドキュメントをひな形から作って」と伝える
4. 生成されたドキュメントを項目ごとに確認・修正する

**作成するファイル**

| ファイル名 | 内容 | 例 |
| ---------- | ---- | -- |
| `product-requirements.md` | 誰のため・何が特徴か | 「先延ばし癖がある人向け。タスクを2分単位に分解するアプリ」 |
| `functional-design.md` | 画面構成・データ設計 | ユースケース図・画面遷移図・データモデルなど |
| `architecture.md` | 使う技術 | 「Next.js + Supabase + Tailwind CSS」 |

**難易度**：⭐⭐☆☆☆

## PHASE2：データベース連携（DAY1 夕方）

> データベースとは？
> アプリが「覚えておく」場所です。ページをリロードしてもデータが消えないのは、データベースに保存されているからです。

> Supabaseとは？
> 本格的なデータベースを無料で簡単に使えるサービスです。ダッシュボードからテーブルの作成やデータの確認ができます。

**目標**：タスクをSupabaseに保存し、リロードしても消えないようにする

**進め方**：

1. `.steering/phase2-database/` フォルダを作る
2. [supabase.com](https://supabase.com) でアカウント作成・プロジェクト作成
3. Supabaseのダッシュボードで `tasks` テーブルを作成する
4. SupabaseのAPIキーを `.env.local` に設定する
5. Claude Code に「Supabaseと繋いで、タスクを保存・取得できるようにして」と伝える
6. 動作確認（リロードしてもデータが残るか）

**成果物**：
- `app/api/tasks/route.ts`（APIの処理）
- `.env.local`（Supabaseの接続情報）

**難易度**：⭐⭐⭐⭐☆

**振り返りチェック**：
- ページをリロードしてもタスクが消えないか？
- Supabaseのダッシュボードにデータが保存されているか？

## PHASE2.5：ログイン機能（DAY2 午前）

> 認証（ログイン機能）とは？
> 「あなたは誰ですか？」を確認する仕組みです。認証があることで「自分のタスクだけ見える」「他人のタスクは見えない」が実現できます。

> RLS（Row Level Security）とは？
> 「ログイン中の自分のデータだけ取得できる」をデータベース側で強制するルールです。Supabaseを使うときは必ず設定が必要です。

> middleware.ts とは？
> アプリへのアクセスを「玄関で見張る係」のファイルです。このファイルがないと、ページ移動のたびにログアウト状態になってしまいます。

**目標**：メールアドレスとパスワードでログイン・サインアップできるようにする

**進め方**：

1. `.steering/phase2-auth/` フォルダを作る
2. Supabaseの Authentication 設定を確認する
3. Claude Code に「ログイン画面を作って、未ログインのときはログイン画面にリダイレクトして」と伝える
4. Claude Code に「middleware.ts を作成して、RLSポリシーも設定して」と伝える
5. ログイン → タスク表示の流れを確認する

**成果物**：
- `app/auth/page.tsx`（ログイン・サインアップ画面）
- `middleware.ts`（ログイン状態を全ページで自動チェック）
- SupabaseのRLSポリシー

**よくあるエラー**：
- `permission denied for table tasks` → RLSポリシーが未設定。Claude Codeに「RLSポリシーを設定して」と伝える
- ページ遷移のたびにログアウトされる → `middleware.ts` が未作成

**難易度**：⭐⭐⭐⭐☆

**振り返りチェック**：
- ログイン・ログアウトの動作を確認できたか？
- 別のアカウントでログインしたとき、データが分離されているか？

## PHASE3：デプロイ（DAY2 午後）

> デプロイとは？
> 自分のパソコンだけで動いていたアプリを、インターネット上に公開することです。デプロイすると、世界中の人がURLからアクセスできます。

> Vercelとは？
> GitHubと連携してWebアプリを自動でインターネットに公開するサービスです。GitHubにコードをアップするだけで、自動でURLが発行されます。

> `.gitignore` とは？
> GitHubにアップしてはいけないファイルを指定するリストです。`.env.local`（APIキー入り）が含まれていることを必ず確認してください。

**目標**：インターネットで公開されるURLを発行する

**進め方**：

1. `.steering/phase3-deploy/` フォルダを作る
2. **`.gitignore` に `.env.local` が含まれているか確認する（重要）**
3. GitHubにリポジトリを作成してコードをアップロードする
4. [vercel.com](https://vercel.com) でGitHubと連携してデプロイする
5. **Vercelの環境変数設定画面で `.env.local` の内容を入力する**（これをしないと本番環境でDB・AI機能が動きません）
6. 発行されたURLをスマホで確認する

**成果物**：世界中からアクセスできるURL（例：`https://task-app-xxx.vercel.app`）

**よくあるミス**：
- Vercelに環境変数を設定し忘れる → DB接続・AI機能が動かない原因
- `.env.local` をGitHubにアップしてしまう → APIキーが漏れる

**難易度**：⭐⭐⭐☆☆

**振り返りチェック**：
- スマホからURLにアクセスできたか？
- ログインしてタスクを追加できたか？

## PHASE4：オリジナル機能追加（DAY2 夕方）

> このフェーズについて
> PHASE1〜3で「動くアプリ」が完成しました。PHASE4では、設計ドキュメントで決めた自分だけの差別化要素を実際に実装します。

**目標**：設計ドキュメントで決めたオリジナル機能を追加して、自分だけのアプリに仕上げる

**進め方**：

1. `.steering/phase4-ai-feature/` フォルダを作る
2. `docs/product-requirements.md` を開いて差別化要素を再確認する
3. Claude Code に「この機能を追加したい」と伝えて実装する
4. GitHubにpushしてVercelに自動反映させる
5. 完成したURLをシェアする

**壁打ちの例**：
```
「私のアプリには『タスクを2分単位に分解する機能』を追加したいです。
どう実装すればいいか一緒に考えてください。」
```

**成果物**：自分だけのオリジナル機能が入った公開済みWebアプリ

**難易度**：⭐⭐⭐☆☆（追加機能の複雑さによる）

**振り返りチェック**：
- 設計ドキュメントで決めた機能が実装できたか？
- 友人・家族に見せたいと思えるアプリになったか？
- GitHubにpushしてVercelに自動反映されたか？
